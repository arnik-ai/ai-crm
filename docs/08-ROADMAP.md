# نقشه‌راه و استراتژی مقیاس‌پذیری

> نسخه ۱.۰ | MVP → Production → Scale (100k+ دانشجو)

## ۱. نقشه‌راه MVP (۶–۸ هفته)
| فاز | خروجی |
|---|---|
| هفته ۱ | اسکلت بک‌اند، دیتابیس، Auth/RBAC، مهاجرت‌ها |
| هفته ۲ | ماژول CRM (دانشجو/دوره/تگ/یادداشت/مراحل فروش) + API |
| هفته ۳ | یکپارچه‌سازی Workano (Webhook + Provider + ثبت تماس) |
| هفته ۴ | یکپارچه‌سازی AvalAI (Whisper + GPT-5.5) + Celery |
| هفته ۵ | LangGraph (استخراج، امتیاز، مرحله، پیگیری، خلاصه) |
| هفته ۶ | داشبورد فرانت (Next.js) + کارت‌ها + قیف فروش |
| هفته ۷ | دستیار چت CRM + پیگیری‌ها |
| هفته ۸ | امنیت، CI/CD، استقرار Parspack، تست UAT |

**معیار پذیرش MVP:** یک تماس واقعی از Workano → ثبت، رونویسی، استخراج، امتیاز و نمایش در داشبورد به‌صورت end-to-end.

## ۲. نقشه‌راه Production
- چندمستأجری کامل (Multi-Tenant) با جداسازی داده.
- مراحل فروش سفارشی per-tenant + اتوماسیون قوانین.
- WebSocket Realtime برای داشبورد.
- گزارش‌گیری پیشرفته + Export (Excel/PDF).
- نقش‌ها و مجوزهای ریزدانه (Fine-grained RBAC).
- مشاهده‌پذیری: tracing، متریک، هشدار.
- تست بار و سخت‌سازی امنیتی (Pentest).

## ۳. استراتژی مقیاس‌پذیری برای ۱۰۰٬۰۰۰+ دانشجو

### ۳.۱ پایگاه‌داده
- **Partitioning** جداول پرحجم (`calls`, `audit_logs`, `activities`) بر اساس زمان.
- **Read Replica** برای داشبورد و دستیار AI (جداسازی خواندن از نوشتن).
- **Materialized Views** برای KPIها با Refresh دوره‌ای.
- ایندکس‌گذاری دقیق + `pg_stat_statements` برای کوئری‌های گران.

### ۳.۲ پردازش AI
- صف‌های Celery مجزا: `telephony`, `stt`, `llm` با اولویت و autoscale مستقل.
- **Backpressure**: محدودسازی همزمانی فراخوانی AvalAI؛ batching استخراج.
- **Cache** نتایج STT/LLM بر اساس hash؛ کاهش هزینه و تأخیر.

### ۳.۳ اپلیکیشن
- API بدون state (افقی‌مقیاس‌پذیر)؛ Session/Token در Redis.
- جداسازی آینده‌ی ماژول‌های پرترافیک (telephony/ai_analysis) به سرویس مستقل (مرزها از قبل آماده‌اند).
- CDN برای فرانت؛ کش پاسخ‌های داشبورد (TTL کوتاه).

### ۳.۴ ذخیره‌سازی
- فایل‌های ضبط در Object Storage با lifecycle policy (انتقال فایل‌های قدیمی به ذخیره‌ی ارزان).

### ۳.۵ مسیر مهاجرت Modular Monolith → Microservices
چون هر ماژول دارای مرز مشخص (domain/application/infrastructure/api) و ارتباط از طریق اینترفیس است،
جداسازی به سرویس مستقل تنها نیازمند جایگزینی فراخوانی in-process با فراخوانی شبکه (REST/Queue) است—
بدون بازنویسی منطق دامنه.

## ۴. شاخص‌های موفقیت (KPIs)
- نرخ تبدیل سرنخ به ثبت‌نام (Conversion Rate)
- زمان پاسخ به پیگیری (Follow-up SLA)
- دقت استخراج اطلاعات AI (با نمونه‌گیری دستی)
- تأخیر پردازش تماس (از Recording Ready تا نمایش تحلیل)
